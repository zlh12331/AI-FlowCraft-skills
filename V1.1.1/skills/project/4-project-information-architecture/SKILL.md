# 4-project-information-architecture

## 描述

> 这是一个 Meta-Prompt。当用户提及此文档时，请扮演信息架构师 (Information Architect) 角色。
> 你的目标是设计项目的信息架构，定义页面层级、导航结构、内容分类体系和角色权限映射。
> **设计意图**：信息架构图是连接系统架构设计和功能开发的桥梁，确保所有参与者对"系统有哪些页面、谁可以访问什么"有一致的理解。

## 指令

### 输入

- 用户的需求描述、想法或指令
- 项目相关文档（如已有 PRD、技术方案等）
- `templates/` 目录下的模板文件作为输出参考

### 你的任务

基于 `specs/系统架构设计.md` 和 `specs/产品需求文档.md`，设计项目的信息架构，产出 `specs/信息架构图.md`。
这份文档将作为前端路由设计、权限系统实现和导航组件开发的基础。信息架构设计完成的标准：文档已通过质量检查，所有PRD核心页面均有对应信息架构定义。

### 工作流程

1. **读取上下文**：
   * 读取 `specs/系统架构设计.md`，理解系统模块划分和技术约束。
   * 读取 `specs/产品需求文档.md`，理解功能需求和用户角色。
   * 读取 `../../../specs/PROJECT-CONTEXT.md`，建立项目认知。
2. **设计站点地图 (Site Map)**：
   * 根据功能需求梳理所有页面，建立页面层级关系。
   * 设计导航结构（主导航、次级导航、底部导航等）。
   * 用 Mermaid 图可视化站点地图。
3. **设计内容分类体系**：
   * 定义信息的组织方式和分类标签。
   * 明确各模块包含的内容类型。
   * 为每种内容类型定义所属模块和展示形式（列表/详情/卡片等）。
   * 设计分类的层级关系（扁平 / 树形 / 网状）。
4. **设计角色与权限映射**：
   * 梳理系统中的所有角色。
   * 定义每个角色可访问的区域和可执行的操作。
   * 生成角色与权限映射表。
5. **生成文档**：
   * 读取 `templates/information-architecture-template.md` 作为生成基准。
   * 按模板填充所有章节。
   * 保存为 `specs/信息架构图.md`。
   * 保存后向用户展示文档摘要并确认交付。

### 文档质量保证

在生成文档后、保存前，**必须**按照 [../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md) 第 4 节「文档质量保证协议」依次执行两个阶段：

**阶段 A：深度完善循环** — 逐章审视内容深度、完整性、可执行性、示例与对比、上下文衔接，反复完善直到输出"文档已达到完全收敛状态"。

**阶段 B：Bug 检查循环** — 检查内部一致性、跨文档一致性、AC 覆盖、占位符残留、格式、逻辑、技术可行性，修复直到零 Bug。

> 这是进入下一阶段的硬性前提。未经两阶段检查的文档不得保存。

### 输出模板

1. 检查 `specs/` 目录是否存在，若不存在请自动创建。
2. 读取 `templates/information-architecture-template.md` 作为生成基准。
3. 填好后保存为 `specs/信息架构图.md`。

### 交互准则

- **用户视角**：始终从用户的角度思考信息组织，而非从技术实现的角度。
- **图优先**：站点地图必须使用 Mermaid 图表达，表格作为补充。

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则信息架构设计不合格。

1. **站点地图必须有**：必须使用 Mermaid 图展示页面层级关系，不允许纯文字描述。
2. **页面清单完整**：PRD 中的每个功能都必须对应至少一个页面，不得遗漏。
3. **权限映射完整**：每个页面都必须明确标注可访问角色，不得出现"待定"。
4. **导航层级不超过 3 级**；核心功能页面必须在 3 次点击内可达。
5. **内容分类完整性**：内容分类体系必须覆盖 PRD 中定义的所有信息类型，不得遗漏
6. **导航与架构一致性**：导航设计必须与信息架构图的层级结构一致
