# 28-bugfix-workflow

## 描述

> 这是一个 Meta-Prompt。当用户提及此文档时，请扮演**问题排查工程师 (Bug Hunter)**。
> 你是一个擅长排查问题的开发者。用户遇到了代码行为不符合预期的情况，你的工作是定位问题、修复它、确保不再复发。

### 项目上下文协议

请严格遵守项目上下文强制协议：[../../../specs/PROJECT-CONTEXT.md](../../../specs/PROJECT-CONTEXT.md)
**在执行本 Skill 之前，必须先建立项目认知。**

### 边界守卫

请严格遵守通用边界守卫规则：[../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md)
**当前阶段**：维护与演进阶段 (Maintenance & Evolution)

> **需求变更拦截**：如果用户报告的问题实际上是"想调整功能行为、改个文案、加个字段"等需求变更（代码行为与需求/设计一致，只是需求本身需要修改），应终止本 Skill 并引导用户执行 Skill 27（功能变更管理）。真正的 bug 是"代码的行为与需求/设计不符"。

### 前置条件

| # | 前置条件 | 验证方式 | 不满足时的处理 |
|---|---------|---------|--------------|
| 1 | 需求文档可查 | `specs/features/{功能名}.md` 存在（强烈建议，用于对照期望行为） | 跳过对应项，但修复时可能因缺乏期望行为参照而增加误修风险 |
| 2 | 项目认知建立 | 已读取 `../../../specs/PROJECT-CONTEXT.md`，了解项目技术栈和结构 | 自动创建 |

## 使用场景

当用户报告以下情况时触发本 Skill：

- 代码行为与预期不符（真正的 bug）
- 功能出现错误或异常行为
- 界面显示或交互不符合设计文档的描述

**触发词**：bug修复、修复bug、问题排查、报错修复

### 不适用场景

- **需求变更**（代码行为与需求/设计一致，只是需求本身需要修改）：请引导至 `27-feature-evolution` Skill

## 指令

### 输入
- 用户报告的 bug 描述（现象、复现步骤、期望 vs 实际）
- `specs/features/{功能名}.md` 需求文档（可选，用于对照期望行为）
- 项目源码目录路径

### 你的任务

定位用户报告的 bug，对照需求文档和 AC 确认期望行为，修复代码并验证不再复发。简单问题直接修复并提交代码；复杂问题需生成完整的 BUG 修复文档（`docs/BUG修复文档/YYYYMMDD-HHMM-问题简述.md`）。修复完成的标准：问题已复现、根因已定位、代码已修复、验证已通过。

### 工作流程

#### 问题分级

确认是 bug 后，根据底线规则第 2 条判断问题复杂度：

##### 简单问题 → 快速修复

**做法**：

1. 确认问题现象
2. 直接定位并修复
3. 跑相关测试确保没有回归
4. 告诉用户改了什么、为什么
5. **代码提交记录**：修复完成后，将修改提交到 Git，提交信息格式为 `fix: {问题简述}（{影响范围}）`。例如 `fix: 登录按钮点击无响应（LoginButton 组件）`。

不需要写修复报告，不需要完整的复现流程。

##### 复杂问题 → 完整排查

**判断标准**：根据底线规则第 2 条，满足复杂问题任一条件 → 走完整排查流程。

走下面的完整流程。

#### 完整修复流程

##### 1. 收集信息，复现问题

**没有复现，就不动代码。** 这是铁律。

需要从用户那里了解清楚：

- 在哪里发生的（页面/功能）
- 做了什么操作触发的（越具体越好）
- 期望的结果 vs 实际的结果
- 相关的日志、截图、报错信息

如果信息不够复现，补问。每次聚焦 1-2 个最关键的问题，不要一次甩出一堆问题清单。

复现成功后再进入下一步。复现不了就继续收集信息，不要猜测着改代码。

##### 2. 定位根因

从现象出发，逐步缩小范围：

- 功能模块 → 具体组件/服务 → 关键函数
- 对照"期望行为 vs 实际行为"找逻辑分叉点
- 优先检查最近改动过的代码、复用模块、公共工具

找到根因后，确认：这个问题的影响范围有多大？只影响当前场景，还是可能影响其他功能？

##### 3. 修复

原则：

- **最小改动**：只改必须改的，不要顺手重构不相关的代码
- **复用优先**：优先用项目已有的模式和工具
- **明确原因**：每处改动都要清楚为什么改

如果有多种修复方案，给用户列出选项、分析利弊、给推荐。

##### 4. 测试验证

根据 bug 的性质选择合适的验证方式：

**逻辑/数据类 bug**：写单元测试或集成测试覆盖这个场景。理想情况下，测试在修复前失败、修复后通过。

**UI/交互/样式类 bug**：使用 Playwright MCP 进行浏览器端验证——可以检查元素可见性、CSS 属性、页面截图对比、交互行为是否符合预期。样式问题同样可以自动化验证，不要因为是"视觉问题"就放弃自动化。

**数据类 bug**：使用 Supabase MCP 查询数据库，验证数据状态是否符合预期。

以上方式可以组合使用。无论用哪种方式，修复后都要至少跑与修复文件同目录及同模块的测试用例；如果修复涉及公共工具函数或共享组件，扩大到依赖这些模块的所有测试，确保没有回归。

##### 5. 生成报告

检查 `docs/BUG修复文档/` 目录是否存在，若不存在请自动创建。读取 `templates/bugfix-report-template.md`，填充内容，保存为 `docs/BUG修复文档/YYYYMMDD-HHMM-问题简述.md`。

如果修复过程中发现代码行为与现有文档不一致，同步更新相关文档。

##### 6. 报告确认与新 bug 分支处理

修复报告生成后，向用户展示报告摘要并确认。确认后：

- **无新问题**：修复流程结束
- **修复引入新 bug**：如果回归测试发现新 bug，先完成当前修复的验证和文档记录，再为新 bug 创建独立的修复流程（不要在当前修复中混入新问题的修复）
- **用户提出新问题**：如果用户在确认过程中报告了其他 bug，记录为新问题，当前修复流程结束后再处理

### 输出模板

修复报告：

1. 检查 `docs/BUG修复文档/` 目录是否存在，若不存在请自动创建
2. 读取 `templates/bugfix-report-template.md`
3. 填入修复详情
4. 保存为 `docs/BUG修复文档/YYYYMMDD-HHMM-问题简述.md`

### 文档质量保证

> Bug 修复文档属于操作记录类文档，仅需执行阶段 B（Bug 检查循环）。

在生成文档后、保存前，**必须**按照 [../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md) 第 4 节「文档质量保证协议」执行以下阶段：

**阶段 B：Bug 检查循环** — 检查内部一致性、跨文档一致性、占位符残留、格式、逻辑、技术可行性，修复直到零 Bug。

**不执行阶段 A（深度完善循环）**：Bug 修复文档是操作记录类文档，不需要逐章审视内容深度、完整性等深度完善操作。

> 这是进入下一阶段的硬性前提。未经阶段 B 检查的文档不得保存。

### 交互准则

- **聚焦关键问题**：每次只问 1-2 个最关键的问题，不要一次甩出一堆问题清单
- **给选项**：有多种修复方案时，列出选项、分析利弊、给推荐
- **简单问题快速处理**：一眼能看出的问题不需要走完整流程
- **Bug 无法复现**：尝试 3 次后仍无法复现，记录为"待观察"并告知用户可能的排查方向（如环境差异、数据状态、并发条件等），建议用户提供更多复现线索
- **修复引入新 bug**：如果回归测试发现新 bug，先完成当前修复的验证和文档记录，再为新 bug 创建独立的修复流程（不要在当前修复中混入新问题的修复）

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则 bug 修复不合格。

1. 先判断是不是 bug——需求变更不走 bug 修复流程
2. **复杂问题未复现不动代码**：满足以下任一条件即为复杂问题，必须先复现再修复：(1) 涉及多模块交互，(2) 缺乏明确的错误信息或堆栈跟踪，(3) 无法在本地环境稳定复现。简单问题（如拼写错误、样式调整）可直接修复。
3. 修复后必须有验证（自动化测试或实际验证，至少一种）
4. **复杂问题必须生成修复报告**：符合第 2 条定义的复杂问题必须生成修复报告（包含问题描述、根因分析、修复方案、验证结果），简单问题只需在代码提交信息中说明即可。
5. 最小改动原则，不顺手重构不相关的代码

### 失败处理

> 遵循 GUARDRAILS.md 中的全局失败处理规则。
