# 项目上下文强制协议

> **CRITICAL: 这是你对用户的承诺，违反即是背叛**

## 你的职责
用户信任你，把他们的项目交给你。这份信任意味着：

在执行任何 Skill 或任务之前，你**必须**：

1. **读取项目文档**（两层策略，避免上下文溢出）：
    - **核心文档（必读）** — 按以下顺序依次读取，建立项目全局认知：
        1. `specs/产品需求文档.md`（Skill 2）— 理解产品目标与功能范围
        2. `specs/技术栈.md`（Skill 3）— 掌握技术选型与工具链
        3. `specs/系统架构设计.md`（Skill 3）— 理解系统整体架构
        4. `specs/信息架构图.md`（Skill 4）— 理解页面层级与权限映射（纯后端项目可选，文件可能不存在）
        5. `specs/数据模型字段约定.md`（Skill 5）— 统一前后端数据语言
        6. `specs/项目结构.md`（Skill 12）— 掌握目录结构与模块划分
    - **其他文档（选读）** — 根据当前 Skill 的需要，读取相关文档：
        - `specs/交互设计文档.md`（Skill 6）— 页面流转、状态说明、表单验证
        - `specs/数据库设计.md`（Skill 7）— ER 图、表结构、索引
        - `specs/API接口文档.md`（Skill 8）— 接口请求/响应格式、错误码
        - `specs/设计规范.md`（Skill 9）— 色板、字体、间距等视觉规范
        - `specs/前端开发规范.md`（Skill 13）— 代码风格、TypeScript 规范
        - `specs/后端开发规范.md`（Skill 14）— 分层规范、异常处理、日志规范
        - `specs/前后端协作规范.md`（Skill 15）— 接口联调、错误处理、鉴权机制
        - `specs/环境与配置文档.md`（Skill 16）— 环境变量、域名规划、CI/CD
        - `specs/开发路线图.md`（Skill 17）— 功能开发顺序与里程碑
        - `specs/需求讨论记录.md`（Skill 1）— 需求讨论的历史决策
        - `specs/features/` 目录下的功能级文档（按需读取，扁平结构）：
            - `specs/features/{功能名}.md`（Skill 2 复杂项目模式产出）— 功能需求与验收标准
            - `specs/features/{功能名}_后端技术方案.md`（Skill 10）— 后端 API 与业务逻辑
            - `specs/features/{功能名}_前端技术方案.md`（Skill 11）— 组件树与数据流
            - `specs/features/{功能名}_任务规划.md`（Skill 19）— 开发任务清单
            - `specs/features/{功能名}_变更任务_{CR序号}.md`（Skill 27）— 功能变更记录与增量任务
2. **读取开发记录**（按需）：扫描 `docs/开发记录/` 目录，了解当前开发进度和历史决策：
    - 初始化记录（Skill 18）— 项目初始化的执行结果
    - 切片完成记录（Skill 20）— 每个垂直切片的编码完成状态
    - 阶段完成报告（Skill 21）— 所有切片完成后的汇总基线
3. **读取 BUG 修复文档**（按需）：扫描 `docs/BUG修复文档/` 目录，了解已知问题和修复历史。文件命名格式：`YYYYMMDD-HHMM-问题简述.md`
4. **扫描代码结构**：了解项目的目录结构和已有代码
5. **确认理解**：确保你知道项目是做什么的、用什么技术、有什么规范

## 为什么这很重要

> **为什么这很重要**：新窗口中 AI 对项目一无所知。不先建立上下文就开始工作，就是在浪费用户之前的所有规划成果。花费 5 分钟读取文档，可以节省 2 小时的返工时间。

## 你的承诺
作为一个专业的 AI 助手，你承诺：
- ✅ 我不会因为懒惰而跳过上下文建立
- ✅ 我不会假装理解我不知道的东西
- ✅ 我不会在不了解项目的情况下就开始编码
- ✅ 我会像对待自己的项目一样对待用户的项目
- ✅ 我会先读文档，再干活
