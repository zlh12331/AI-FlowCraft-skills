## 你的任务
基于 `specs/系统架构设计.md`、`specs/信息架构图.md` 和 `specs/产品需求文档.md`，定义项目的数据模型和字段约定，产出 `specs/数据模型字段约定.md`。
这份文档将作为数据库设计（7-project-database-design）和 API 设计的基础，确保前后端对数据格式、命名规则和空值处理有一致的理解。数据模型约定完成的标准：文档已通过质量检查，所有核心业务实体均有字段定义。

## 工作流程

1. **读取上下文**：
   * 读取 `specs/系统架构设计.md`，理解技术栈和数据存储方案。
   * 读取 `specs/信息架构图.md`，理解页面结构和内容类型。
   * 读取 `specs/产品需求文档.md`，理解业务需求和数据实体。
   * 读取 `../../../specs/PROJECT-CONTEXT.md`，建立项目认知。
2. **定义枚举值**：
   * 从 PRD 中提取所有状态码、类型码等枚举值。
   * 为每个枚举定义名称、值、说明和使用场景。
   * 确保枚举值在前后端一致使用。
3. **定义特殊字段格式**：
   * 确定时间戳格式（ISO 8601 / Unix 时间戳 / 自定义格式）。
   * 确定金额精度和货币格式。
   * 确定日期、手机号、邮箱等字段的统一格式。
4. **定义空值处理规则**：
   * 明确 null、空字符串、缺失字段各自的语义。
   * 定义前端、后端、数据库三端的空值处理策略。
   * 确保三端行为一致。
5. **定义前后端字段名映射**：
   * 确定命名风格（前端 camelCase / 后端 snake_case）。
   * 定义转换规则和特殊例外。
   * 确保映射规则清晰无歧义。
6. **生成文档**：
   * 读取 `assets/data-model-convention-template.md` 作为生成基准。
   * 定义通用字段约定（ID 格式、时间戳格式、软删除标记、审计字段等）。
   * 按模板填充所有章节。
   * 保存为 `specs/数据模型字段约定.md`。
   * 保存后向用户展示文档摘要并确认交付。

## 交互准则
- **前后端共同视角**：始终同时考虑前端和后端的实现便利性，不偏向任何一方。
- **约定优于配置**：尽量定义统一的规则，减少特殊情况。如果必须有例外，必须在文档中明确标注。

## 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则数据模型约定不合格。

1. **枚举值完整**：PRD 中涉及的所有状态码、类型码都必须定义枚举值，不得遗漏。
2. **空值规则明确**：每种空值场景（null / 空字符串 / 缺失字段）都必须有明确的三端处理策略。
3. **字段名映射无歧义**：前后端字段名映射规则必须清晰，特殊例外必须逐一列出。
4. **金额精度安全**：涉及金额的字段必须使用整数或 Decimal 类型，禁止使用浮点数。
5. **跨文档一致性**：实体名和字段名必须与 PRD 中的术语完全一致；如有命名调整，必须在"字段名映射规则"章节中明确标注
6. **下游兼容性**：所有约定必须可直接被 Skill 6（交互设计）、Skill 7（数据库设计）和 Skill 8（API 设计）引用，不得有歧义或遗漏
