## 你的任务

基于项目的技术栈和业务特点，制定一份**可直接执行的前端开发规范文档**（`specs/前端开发规范.md`）。这份规范将作为团队所有前端开发者的日常工作手册，也是 AI 辅助编码时的行为准则。前端规范完成的标准：`specs/前端开发规范.md` 已保存并通过阶段 B 质量检查，每条规范都配有代码示例。

## 工作流程

### Step 1: 读取上下文

读取前置条件中列出的文档，建立项目全景认知。重点关注：

- 技术选型中的前端框架（React / Vue / Angular / Svelte / 其他）、状态管理方案、路由方案
- 目录结构中的前端部分约定
- API 规范中的请求/响应格式约定
- 数据模型中的前端数据流转方式

### Step 2: 技术栈适配分析

针对前端技术栈进行深度分析，输出适配报告：

| 分析维度 | 分析内容 |
|---------|---------|
| 语言 | TypeScript 版本、严格模式配置、目标 ES 版本 |
| 框架 | React / Vue / Angular / Svelte 版本、核心 API 范围、生态工具链 |
| 样式方案 | CSS Modules / Tailwind / Styled Components / 其他 |
| 包管理器 | pnpm / npm / yarn，monorepo 方案 |
| 构建工具 | Vite / Webpack / Next.js 内置 |
| 代码质量工具 | ESLint / Prettier / Stylelint 配置策略 |
| 测试框架 | Jest / Vitest / Testing Library |
| 团队规模 | 1-3 人 / 4-10 人 / 10+ 人，决定规范严格程度 |

### Step 3: 逐章节生成规范

按照模板中的 9 个章节逐一生成规范内容。每个章节必须：

- 先说明"为什么需要这条规范"（一句话）
- 再给出具体规则（可直接执行）
- 最后附上正确示例和错误示例的对比

### Step 4: 一致性检查

生成完成后，进行自检：

- [ ] 所有示例代码是否与项目技术栈一致（没有用 Vue 示例但项目是 React）
- [ ] 命名约定是否与目录结构规范一致
- [ ] ESLint 规则是否与 Prettier 配置不冲突
- [ ] TypeScript 严格度是否与 tsconfig.json 一致
- [ ] 样式规范是否与选型中的样式方案一致
- [ ] AI 协作协议是否覆盖了常见的 AI 编码陷阱

检查不通过时，回到对应章节补充或修正后重新检查。多次修正仍不通过时，暂停并向用户报告具体问题。

### Step 5: 生成文档
读取 `assets/frontend-standards-template.md` 作为生成基准，将填充完成的模板保存为 `specs/前端开发规范.md`。文档头部需填入当前日期（YYYY-MM-DD 格式）作为 `{{生成时间}}`，从 `specs/技术栈.md` 中提取技术栈概要信息作为 `{{技术栈概要}}`。

* 保存后向用户展示文档摘要并确认交付。

## 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则规范不合格。

1. **组件命名必须遵循 PascalCase**，样式文件使用 kebab-case，工具函数使用 camelCase。
2. **每条规范必须配有可直接执行的代码示例**，不允许纯文字描述的抽象规则。
3. **规范必须与项目技术栈一致**，示例代码必须使用项目实际选型的框架和工具链，不得用 Vue 示例但项目是 React。
4. **AI 协作协议必须包含错题本机制**，记录 AI 常见的编码陷阱和正确做法，减少重复犯错。
5. **样式方案必须唯一。** 模板第 6 章中必须明确指定唯一样式方案并列出禁止使用的替代方案，不允许留空或标注"均可"。各样式方案的专属规范要点如下（仅对应方案的项目适用）：
   - **CSS Modules 项目**：必须使用 `*.module.css` 命名约定，使用 camelCase 类名，禁止使用全局选择器，组件间样式复用使用 `compose`。
   - **Styled Components 项目**：必须使用 `styled.<element>` 或 `styled(Component)` 创建组件，主题注入使用 `ThemeProvider`，全局样式使用 `createGlobalStyle`。
   - **TailwindCSS 项目**：必须使用 Design Token（CSS 变量）扩展默认主题，禁止直接使用 Tailwind 默认颜色类名。
6. **TypeScript 项目严格模式必须开启**（仅 TypeScript 项目适用）：`tsconfig.json` 中 `strict: true` 必须开启，`noImplicitAny` 不得设为 false。禁止在组件 Props 或 API 响应类型中使用 `any`（如需使用必须：(1) 注释说明原因 (2) 创建 TODO 标注计划修复时间。建议优先使用 `unknown` 替代）。此规则与 Skill 14 规则 9、Skill 10 规则 11 对齐。
7. **组件目录结构必须与项目结构一致**：组件的存放位置必须遵循 `specs/项目结构.md` 中定义的目录约定（如 `src/components/`、`src/features/`），不得随意创建新目录层级。公共组件与业务组件必须分离存放。
8. **状态管理方案必须唯一且规范必须覆盖使用模式**：根据项目技术栈选择对应的状态管理方案。React 项目推荐 Zustand 或 Redux；Vue 3 项目推荐 Pinia；Angular 项目推荐 NgRx 或 Signals。使用 NgRx 时，规范中必须说明 Action 类型命名（[Source] Event 格式）、Reducer 纯函数要求、Selector 使用 memoization、Effect 处理副作用（仅 Angular 项目适用）；Svelte 项目推荐内置 Store（`svelte/store`）。规范中必须说明 store 的命名规则、存放位置（如 `src/stores/`）、导出方式（named export）。不允许混用多种状态管理方案。
9. **API 请求层必须统一封装**：规范中必须定义统一的 HTTP 客户端封装（如 Axios instance），包括 baseURL 配置、请求/响应拦截器、Token 注入、错误统一处理。禁止在组件中直接调用 `fetch` 或创建新的 Axios 实例。
