### 你的任务

读取任务规划，按 RED-GREEN-REFACTOR 循环执行开发任务。每个任务必须经过 TDD 循环和验收测试两个阶段才能标记完成。产出代码和 `docs/开发记录/{功能名}_切片{N}_完成记录.md`。

读取文档后，重点理解：

- 需求文档中的验收标准（AC）——这是测试的最终依据
- 技术方案中的 API 定义、数据库设计、核心逻辑——这是实现的依据
- 任务清单中当前阶段的任务列表、验证标准、依赖关系——这是执行的依据

切片完成的标准：TDD循环（RED→GREEN→REFACTOR）已完成，验收测试通过，完成记录已生成。

### 工作流程

#### 总览步骤序列

```text
开始阶段 → 确认依赖完成 → 创建 Task → 执行任务（TDD 循环 + 验收测试）→ 标记完成 → 下一个任务 → ... → 完成阶段 → 生成报告
```

**异常路径**：验收重试 3 次仍失败 → 向用户报告 → 用户选择继续/回退/暂停

#### 一个任务什么时候算"完成"

**TDD 循环通过 ≠ 任务完成。**

TDD 验证的是代码逻辑正确，但代码逻辑正确不代表功能在真实环境中表现正确——组件测试通过不代表页面整体渲染正常，单元测试通过不代表数据真的写进了数据库。

一个任务标记完成，需要同时满足两个条件：

1. **TDD 循环通过**（代码逻辑正确）
2. **验收测试通过**（真实环境中功能符合预期）

#### TDD 循环

##### RED — 先写测试，确认失败

根据 Skill 19 任务规划中的**验证标准**和对应的**AC**编写测试。测试要覆盖正常情况、边界情况、异常情况。每个测试只验证一个行为。

写完后运行测试，**当前任务新写的测试必须全部失败**（之前任务的测试应保持通过状态）。失败原因应该是"功能未实现"——如果是语法错误或环境问题，先修测试本身。

如果测试没写实现就通过了，说明测试没有真正验证功能，或者功能已经被其他代码实现了。两种情况都需要重新评估。

##### GREEN — 写最少的代码让测试通过

严格按技术方案实现。只写让当前测试通过所需的代码，不多写。优先复用项目中已有的代码和模式。

运行测试（包括之前任务的测试，防止回归），**必须全部通过**。如果有测试失败，修实现代码，不要改测试。但如果发现测试本身有错误（如断言逻辑不对），应先修复测试再继续。

##### REFACTOR — 在测试保护下优化

检查有没有重复代码可以提取、命名是否清晰、结构是否合理、代码质量是否好。每次改动后跑测试，**必须始终通过**。如果测试挂了，回退改动。

简单任务可能不需要重构，这没问题。不要为了"走完流程"而强行重构。

#### 验收测试

TDD 循环通过后，根据任务性质判断需要什么验收方式：

**有 UI 变化的任务**：**必须**用 Playwright MCP 在实际页面上验证 AC 描述的用户操作路径。关键验证点截图留档。TDD 的单元测试/组件测试不能替代浏览器端验收——组件渲染正常不代表页面整体表现正确。

**纯后端/工具函数任务**：TDD 阶段的单元测试已覆盖核心逻辑，验收自然通过，无需额外操作。

**涉及数据库变更的任务**：用 Supabase MCP（仅使用 Supabase 的项目适用）查询数据库，验证数据状态是否符合预期。可与上面两种方式组合。

**怎么判断**：任务涉及用户能看到的变化（页面、组件、样式、交互）→ 有 UI 变化，必须走浏览器端验收。

验收未通过时，进入修复循环：分析原因 → 修复代码 → 重跑单元测试（防回归）→ 重跑验收。**最多重试 3 次**，仍然失败就向用户报告具体原因和已尝试的修复。报告后提供以下可选处理路径：

- **路径 A：继续尝试** — 用户认为可以修复，给出新的修复方向，继续重试（路径 A 最多追加 3 次，累计超过 6 次仍失败则强制进入路径 B 或 C）
- **路径 B：回退代码** — 撤销本次任务的代码变更，回到任务开始前的状态，记录问题供后续分析
- **路径 C：暂停并升级** — 暂停当前阶段，通过 Skill 27（功能变更）或 Skill 28（Bug 修复）走专项流程处理

#### 推进方式

整个阶段的执行过程使用 Task 工具（TaskCreate / TaskUpdate）进行规划和追踪。

##### 开始阶段前

确认当前阶段的所有依赖任务已完成（任务规划中已勾选）。如果有未完成的依赖，告诉用户需要先完成哪些任务。

为当前阶段的每个任务创建 Task，然后开始执行。不需要逐个任务都先停下来问"是否继续"——直接做，遇到需要决策的点再问。

**Mock 数据准备**：如果任务规划中标注了需要 Mock 对接（如前端开发时后端 API 尚未就绪），在开始编码前先创建 Mock 数据/服务，确保前端可以独立开发和测试。Mock 数据的结构必须与 API 接口文档中定义的响应格式一致。

##### 执行任务

按任务顺序逐个执行。每个任务都必须走完上面的**两个阶段**（TDD 循环 + 验收测试）才能标记完成。不要 TDD 通过就直接进入收尾——先判断这个任务需要什么类型的验收，做完验收再继续。

过程中自然地向用户展示关键信息：

- 写了什么测试、测试了什么行为
- 测试失败/通过的结果
- 实现的核心逻辑
- 重构了什么（如果有）
- 验收测试的方式和结果

展示的目的是让用户跟上进度，不是做格式化汇报。根据任务的复杂度调整展示的详细程度——简单任务简要说明即可，复杂任务多展示一些关键决策。

##### 完成阶段后

1. 跑一遍当前阶段所有任务的测试，确保没有回归
2. 对照 AC 检查这个阶段覆盖了哪些验收标准
3. 在任务规划文档中把已完成的任务标记为 `[x]`，确认所有 Task 状态已更新
4. 如果实现过程中发现代码行为与文档描述不一致，同步更新文档
5. 读取 `templates/slice-completion-report-template.md`，生成切片完成记录，保存到 `docs/开发记录/{功能名}_切片{N}_完成记录.md`

> **切片编号说明**：切片编号 {N} 对应任务规划文档（`specs/features/{功能名}_任务规划.md`）中阶段的顺序编号。

##### 所有阶段完成后

当任务规划中的所有阶段（切片）均已完成时，执行功能级最终验收：

1. 逐条对照需求文档（`specs/features/{功能名}.md`）中的所有 AC，确认每条 AC 均已通过
2. 如有未通过的 AC，记录为遗留问题并通知用户
3. 提示用户执行 Skill 21（阶段完成报告）生成汇总基线

#### 技术方案偏差处理

在编码过程中，如果发现技术方案描述的方案实际不可行（如某个库不支持所需功能、数据库查询性能不达标、API 设计与实际需求矛盾等）：

1. **立即暂停编码**：不要自行修改方案继续编码
2. **记录偏差**：在切片完成记录中标注 `⚠️ 技术偏差：[具体描述]`
3. **提出修正建议**：给出至少一个可行的替代方案
4. **等待用户确认**：将偏差和建议展示给用户，由用户决定：
   - **继续**：采用替代方案继续编码
   - **修正技术方案**（适用场景：技术方案本身有误但需求不变）：暂停当前阶段，通过 Skill 10/11 重新生成技术方案，再通过 Skill 19 更新任务规划后继续。**注意：如果偏差源于需求变更（而非纯技术方案问题），应优先选择路径 3**
   - **回退**（适用场景：需求或技术方案都需要调整）：暂停当前阶段，通过 Skill 27（功能变更）走变更流程修改技术方案

### 文档质量保证

> 切片完成记录属于操作日志类文档，仅需执行阶段 B（Bug 检查循环）。

在生成文档后、保存前，**必须**按照 [../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md) 第 4 节「文档质量保证协议」执行以下阶段：

**阶段 B：Bug 检查循环** — 检查内部一致性、跨文档一致性、占位符残留、格式、逻辑、技术可行性，修复直到零 Bug。

**不执行阶段 A（深度完善循环）**：切片完成记录是操作日志类文档，不需要逐章审视内容深度、完整性等深度完善操作。

> 这是进入下一阶段的硬性前提。未经阶段 B 检查的文档不得保存。

### 交互准则

- **直接执行**：不需要逐个任务都先停下来问"是否继续"——直接做，遇到需要决策的点再问
- **进度透明**：自然地展示关键信息（测试结果、实现逻辑、验收方式），但不做格式化汇报
- **如实说明**：遇到问题如实说明，不要隐瞒或假设用户已知情
- **MCP 工具不可用**：如果 Playwright MCP 或 Supabase MCP 不可用，跳过浏览器端/数据库端验收，在切片完成记录中标注"跳过原因"，建议用户手动验证
- **TDD RED 阶段跳过提醒**：用户要求跳过 TDD RED 阶段时，提醒用户 RED 阶段的价值（确保测试真正验证功能而非假通过），但最终尊重用户决定

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则实现不合格。

1. **没有失败的测试，就不写实现代码。** 这是 TDD 的铁律。以下场景可豁免 TDD RED 阶段：(1) 纯配置文件编写；(2) 数据库迁移脚本；(3) 纯类型定义文件。豁免时在完成记录中标注原因。如果用户明确要求跳过 RED 阶段，按交互准则第 5 条处理，并在完成记录中标注跳过原因
2. **TDD 循环通过 ≠ 任务完成。** 有 UI 变化的任务必须经过浏览器端验收测试才能标记完成
3. 不写超出当前测试范围的代码——只写让测试通过所需的最少实现
4. 重构不改行为——测试必须始终通过。简单任务（如纯数据展示）可跳过重构阶段
5. **严格按任务规划执行**：每次只执行用户指定的阶段（切片），不跳过任务、不超前执行、不做计划外的事。用户明确要求跳过某个任务的除外。
6. 优先复用项目已有的代码、组件、模式，不另起一套
7. **遇到问题必须立即暂停并报告**：遇到以下情况必须暂停并向用户报告：(1) 测试无法通过且原因不明，(2) 技术方案与实际实现存在偏差，(3) 依赖的 API/服务不可用，(4) 代码行为与需求/设计不符（即 bug），应记录当前进度后切换到 Skill 28 走 bug 修复流程。报告中必须包含：问题描述、已尝试的解决方法、建议的下一步。暂停后按'技术方案偏差处理'流程执行，等待用户决策后再继续或回退，不视为违反规则 5。
8. **TDD 阶段必须记录**：每个任务的完成记录中必须包含 TDD 执行记录表（RED/GREEN/REFACTOR 三个阶段），记录每个阶段的操作和测试结果。不允许只展示最终结果而省略过程性记录。
9. **Route Handler 必须有测试**：涉及 API 路由（route.ts）（仅 Next.js App Router 项目适用）的任务必须有独立的集成测试或使用 `vi.mock()`（仅 Vitest 项目；Jest 项目使用 jest.mock()）模拟 Request 对象的单元测试。不允许使用"间接覆盖"作为跳过 Route Handler 测试的理由。
10. **API 一致性必须检查**：每个切片完成记录中必须包含"API 一致性检查"章节，逐项对比 API 接口文档与实际实现的路径、响应结构、错误格式，发现不一致必须记录为设计偏差。
11. **跨文档一致性自动校验**：每个切片完成后，执行 `python specs/validate_consistency.py --specs-dir specs/ --check api-paths` 校验 API 路径一致性。如果校验结果包含 FAIL，必须修复不一致问题后才能标记切片完成。
    > 如果脚本不存在（如 `specs/validate_consistency.py` 未部署），则手动逐项比对 API 接口文档与实际实现的路径一致性，确保一致。
12. **组件错误处理必须可测试**：组件中的异步操作错误处理必须使用明确的错误类型区分，不能使用 `err.message || '通用消息'` 这种模式（因为 Error 对象的 message 始终有值，通用消息永远不会被使用）。应改为根据错误类型或错误码来决定显示哪个消息，例如：
    - API 返回的业务错误（有 code）：显示 `error.message`
    - 网络错误（无 code）：显示通用消息
    - 这样测试才能通过模拟不同错误类型来验证不同的 UI 行为（仅 JavaScript/TypeScript 项目适用）。
13. **状态管理必须支持外部 mock**：使用状态管理库时（仅 JavaScript/TypeScript 项目适用），Store 的导出方式必须支持在测试中被 mock 替换。推荐使用独立的 store 文件和 named export，避免在组件内部直接创建 store。例如：
    - ✅ `export const useAuthStore = create<AuthState>(...)` （独立文件，named export）
    - ❌ `const useAuthStore = create<AuthState>(...)` （组件内部创建，无法 mock）
    - Redux 项目使用 configureStore 的 reducer 替换（仅 Redux 项目适用）
    - Pinia 项目使用 createPinia + setActivePinia(mockPinia)（仅 Pinia 项目适用）
    - NgRx 项目使用 provideMockStore（仅 Angular/NgRx 项目适用）
    - svelte/store 项目直接 reassign store value（仅 Svelte 项目适用）
14. **新增依赖必须同步声明**：实现功能时如果引入了新的 npm 依赖（如 date-fns、zod 等），必须在对应子项目的 package.json 中添加声明，不能只在使用处 import 而不声明依赖（仅 Node.js 项目适用）。
15. **框架异常处理必须使用正确的参数形式**：使用后端框架的内置异常类时，必须遵循框架的构造函数签名。
    - **NestJS 项目**：`HttpException` 子类（如 `BadRequestException`、`NotFoundException`）必须使用**对象形式**传递自定义错误码和消息。`throw new BadRequestException({ code: 'AUTH_CODE_INVALID', message: '验证码错误' })` ✅；`throw new BadRequestException('AUTH_CODE_INVALID', '验证码错误')` ❌（第二个参数被忽略）。
    - **Express/Fastify 项目**：使用自定义 Error 类或 `http-errors` 库，确保 status code 和 message 正确传递。
    - **Django/FastAPI 项目**：使用框架内置的 `HTTPException`（FastAPI）或自定义异常类。
    - **自检方法**：搜索所有 `throw new` + `Exception` 的代码，检查参数形式是否符合框架规范。
16. **回溯修复已知问题**：执行前必须先读取阶段报告（Skill 21）中的已知问题列表（ISSUE-XXX），优先修复所有阻塞性问题后再开始编码。如果阶段报告尚未生成，应先完成前置 Skill。
17. **API 全量消费**：前端编码必须为 API 设计文档（Skill 8）中定义的每个端点编写对应的 API 调用函数。不允许后端已实现但前端未调用的端点存在。
    > **自动化验证**：编码完成后，必须执行 `scripts/validate-api-alignment.ts` 脚本，对比后端 `routes.ts` 中定义的所有端点与前端 `api/*.ts` 中的请求路径。脚本输出差异报告，如有未对齐的端点，必须修复后才能标记完成。
18. **入口文件更新**：编码完成后，必须更新应用的入口文件（如 `app.ts`、`index.ts`、`main.tsx`），确保所有新创建的模块（路由、Store、页面等）被正确导入和注册。禁止将路由注册代码留在注释状态。
19. **API 路径一致性检查**：编码完成后，必须验证以下一致性：
    - 前端 `apiClient` 的 `baseURL` 必须与后端路由前缀一致（如 `/api/v1`）
    - 前端 API 函数的路径参数必须与后端路由参数一致（如 `boardId`、`taskId`）
    - 前端 API 的请求方法（GET/POST/PUT/PATCH/DELETE）必须与后端一致
    - 如发现不一致，必须修复后才能标记完成
20. **Service 层业务规则逐项对照**：编码完成后，必须读取后端技术方案（Skill 10）中的"Service 层业务规则"章节，逐项对照确认每条业务规则（如乐观锁校验、级联删除策略、唯一性校验、数量限制、权限校验等）都已在代码中实现。对照结果必须记录在切片完成记录的"业务规则实现对照表"中。如有未实现的规则，必须记录原因并报告用户，不得静默跳过。
