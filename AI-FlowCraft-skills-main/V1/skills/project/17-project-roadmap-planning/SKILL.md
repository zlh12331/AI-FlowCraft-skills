# 17-project-roadmap-planning

## 描述

你的目标是基于产品需求和模块依赖，规划合理的开发顺序和里程碑。

### 项目上下文协议

请严格遵守项目上下文强制协议：[../../../specs/PROJECT-CONTEXT.md](../../../specs/PROJECT-CONTEXT.md)

**在执行本 Skill 之前，必须先建立项目认知。**

### 边界守卫

请严格遵守通用边界守卫规则：[../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md)

**当前阶段**：规范与规划阶段 (Standards & Planning)。

### Skill 边界

- 本 Skill 仅负责规划项目级开发路线图（开发顺序、里程碑、模块依赖），不负责：(1) 具体功能的任务拆分——那是 Skill 19（功能任务规划）的工作；(2) 技术方案的设计——那是 Skill 10/11 的工作；(3) 项目结构的设计——那是 Skill 12 的工作；(4) 工期或人天的精确估算——本 Skill 只规划顺序和依赖，不承诺时间。如果用户要求超出路线图规划的范围，应引导至对应 Skill。

## 使用场景

在 Skill 2（PRD）和 Skill 12（项目结构）完成后，当用户需要基于产品概述和模块依赖规划功能的开发顺序和里程碑时触发。典型触发词：**规划路线图、开发顺序、里程碑规划**。

### 不适用场景

- **产品需求文档未就绪**：请先完成 Skill 2 生成 PRD
- **项目结构未定义**：请先完成 Skill 12
- **需要具体功能的任务拆分**：请引导至 Skill 19
- **需要精确工期或人天估算**：本 Skill 只规划顺序和依赖，不承诺时间

## 指令

### 输入

- 用户的需求描述或指令
- 项目相关文档（如已有 PRD、技术方案等）
- `templates/` 目录下的模板文件作为输出参考

### 任务目标

你的目标是解决"先做什么，后做什么"的问题。基于《产品概述》中的核心板块，分析模块间的依赖关系，制定一份合理的**开发路线图 (Development Roadmap)**，即 `specs/开发路线图.md`。

新手开发者往往容易陷入"迷茫"，不知道在项目初始化（`docs/开发记录/初始化记录.md`）完成后，该从哪个功能开始下手。你需要提供一个清晰的导航图。

**输入**：

- `specs/产品需求文档.md` (提取核心板块和业务流程)
- `specs/系统架构设计.md` (提取模块划分和依赖关系)
- `specs/项目结构.md` (参考模块划分)

路线图完成的标准：开发阶段已规划，里程碑和依赖关系已明确，用户已确认。

### 工作流程

1. **依赖分析 (Dependency Analysis)**：
   - 识别哪些模块是"地基"？（通常是：用户/认证、基础配置、公共组件）。
   - 识别哪些模块是"核心业务"？（必须优先完成，否则产品无价值）。
   - 识别哪些模块是"锦上添花"？（可以延后）。

2. **里程碑规划 (Milestone Planning)**：
   - **Milestone 1: MVP (最小可行性产品)** - 包含最核心的业务闭环（如：地基 + 核心业务）。
   - **Milestone 2: 完整版 (Full Features)** - 包含所有主要功能。
   - **Milestone 3: 增强版 (Enhancement)** - 包含非功能性优化、统计报表等。

3. **排序与指令 (Sorting & Commands)**：
   - 为每个里程碑内的功能模块建议开发顺序。
   - 为每个模块生成具体的提示词 (例如：`"开发[具体模块名]..."`)，让 roadmap 具有可执行性。

4. **进度检测 (Progress Detection)**：
   - **扫描现有文件**：
     - 基于 `specs/项目结构.md` 中定义的模块目录路径进行检测，若对应模块目录已存在 → 标记对应模块开发任务为 `[x]`。
   - **验证"已完成"状态**：对于标记为 `[x]` 的模块，进一步验证其完整性——(1) 检查对应的需求文档中 AC 是否有测试覆盖，(2) 检查是否有对应的切片完成记录（`docs/开发记录/` 目录下），(3) 若仅有目录但无测试和完成记录，标记为 `[~]`（部分完成）并提示用户确认。

5. **生成文档**：输出符合模板的 Markdown 文档，**根据检测结果预先勾选已完成的任务**。

6. **最终交付**：保存到 `specs/开发路线图.md`。

### 输出模板

1. 检查 `specs/` 目录是否存在，若不存在请自动创建。
2. 读取 `templates/roadmap-planning-template.md` 作为生成基准。
3. 填好后保存为 `specs/开发路线图.md`。

### 交互准则

- **最终交付**：详见 GUARDRAILS.md 全局交互准则第 7 条（保存路径：`specs/开发路线图.md`）。
- **里程碑分歧处理**：用户对里程碑划分有不同意见时，优先采纳用户意见并记录调整原因。

### 文档质量保证

在生成文档后、保存前，**必须**按照 [../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md) 第 4 节「文档质量保证协议」依次执行两个阶段：

**阶段 A：深度完善循环** — 逐章审视内容深度、完整性、可执行性、示例与对比、上下文衔接，反复完善直到输出"文档已达到完全收敛状态"。

**阶段 B：Bug 检查循环** — 检查内部一致性、跨文档一致性、功能-里程碑映射完整性、占位符残留、格式、逻辑、技术可行性，修复直到零 Bug。

> 这是进入下一阶段的硬性前提。未经两阶段检查的文档不得保存。

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则路线图不合格。

1. **每个里程碑的完成标准必须包含**：(1) 已完成的功能列表（可勾选），(2) 对应的 AC 通过状态，(3) 交付物清单（代码、文档、测试）。
2. **模块依赖关系必须可视化**：必须用 Mermaid 图展示模块间的依赖关系，不允许纯文字描述。
3. **进度检测基于实际文件**：通过扫描项目目录判断已有模块，已存在的标记为 `[x]`，不得凭记忆判断。
4. **开发顺序必须基于依赖分析，禁止主观排序**：地基模块（认证/配置/公共组件）必须在核心业务之前，核心业务必须在增强功能之前。依赖图中不允许出现循环依赖，如发现必须标注并建议解耦方案。
5. **每个功能模块必须关联对应的 specs 文档路径**：路线图中的每个功能模块必须标注其需求文档和技术方案的路径。需求文档位于 `specs/features/{功能名}.md`（Skill 2 产出），技术方案位于 `specs/features/{功能名}_后端技术方案.md`（Skill 10 产出）和 `specs/features/{功能名}_前端技术方案.md`（Skill 11 产出），确保开发者可以直接定位到完整的设计文档。（纯后端项目仅需关联后端技术方案路径）
6. **里程碑之间的交付物必须可独立验证**：每个里程碑完成后，必须能通过运行测试套件验证功能正确性，不得出现"代码写完了但测试还没写"的情况。里程碑交付标准中必须包含"单元测试通过率 100%，集成测试和 E2E 测试按里程碑阶段逐步覆盖"。
7. **路线图必须包含风险提示**：对于技术风险较高（如第三方支付集成、实时通信）或依赖外部服务的模块，必须在路线图中标注风险等级和备选方案。
8. **Phase 1 之前必须包含"规范与基础设施"阶段**：在第一个功能模块开发之前，必须确保 Skill 13（前端规范）、Skill 14（后端规范）、Skill 15（协作规范）、Skill 16（环境配置）已产出对应文档。如未完成，路线图中必须将规范制定列为 Phase 0 任务。（纯后端项目可跳过此要求）
