### 你的任务

基于数据模型字段约定、系统架构设计和产品需求文档，设计完整的数据库方案（`specs/数据库设计.md`），包含 ER 图、表结构定义、索引策略和可直接执行的建表 SQL。数据库设计完成的标准：文档已通过质量检查，所有核心实体均有表结构定义和建表 SQL。

### 工作流程

1.  **读取上下文**：
    *   读取 `specs/数据模型字段约定.md`：了解已约定的字段命名、类型规范。
    *   读取 `specs/系统架构设计.md`：了解系统分层、服务划分和数据流。
    *   读取 `specs/产品需求文档.md`：提取功能需求，反推数据实体和关系。
2.  **编写数据库概述**：
    *   确定数据库基本信息：数据库类型、版本、字符集、排序规则、存储引擎、连接池配置。
        > **数据库类型说明**：
        > - 关系型数据库：PostgreSQL（推荐，功能丰富、扩展性强）、MySQL（成熟稳定、社区活跃）、SQLite（仅轻量级/嵌入式场景适用；不支持并发写入，不适合生产环境高并发场景）
        > - 非关系型数据库：MongoDB（文档型数据库，适用于灵活 Schema 场景；仅 MongoDB 项目适用）、Redis（键值缓存，适用于会话管理/排行榜/热点数据缓存；仅 Redis 项目适用）、Elasticsearch（全文搜索引擎，适用于复杂搜索/日志分析场景；仅 Elasticsearch 项目适用）
    *   提炼设计原则：根据业务场景和系统架构设计，列出本数据库设计的核心原则（如范式与反范式的取舍、一致性要求等）。
    *   整理命名规范：引用 `specs/数据模型字段约定.md` 中的命名规则，确保表名、字段名、索引名、外键名与项目约定一致。
3.  **设计 ER 图**（以下规则主要适用于关系型数据库；MongoDB 项目使用 Collection 文档设计替代）：
    *   从功能需求中提取核心实体及其关系（一对一、一对多、多对多）。
    *   使用 Mermaid erDiagram 表达实体关系。
    *   **NoSQL 替代设计规则**（仅对应数据库项目适用）：
        *   **MongoDB 项目**：Collection 设计需定义文档 Schema（字段名、类型、必填/可选、索引），嵌入文档 vs 引用文档的选择标准为：数据频繁一起查询则嵌入，独立访问则引用（仅 MongoDB 项目适用）
        *   **Redis 项目**：需定义 Key 命名规范（如 `user:{id}:profile`）、数据结构选型（String / Hash / Set / ZSet / List）、TTL 策略、缓存更新/失效策略（仅 Redis 项目适用）
        *   **Elasticsearch 项目**：需定义 Index 命名规范（如 `{service}-{version}`）、Mapping 设计（字段类型、analyzer）、索引分片/副本数配置（仅 Elasticsearch 项目适用）
4.  **定义表结构**（以下规则主要适用于关系型数据库；MongoDB 项目使用 Collection 文档设计替代）：
    *   为每个实体定义完整的字段列表（字段名、类型、约束、默认值、注释）。
    *   定义主键、外键、唯一约束、非空约束。
    *   设计索引策略：主键索引、唯一索引、普通索引、联合索引。
5.  **编写建表 SQL 语句**：
    *   生成可直接执行的 DDL 语句。
    *   包含完整的字段定义、约束和索引。
    *   SQL 语句需符合目标数据库的语法规范。
6.  **设计分库分表策略**（如涉及）：
    *   评估数据量增长趋势，判断是否需要分库分表。
    *   **判断标准**：以下为参考阈值（以 MySQL 为例，PostgreSQL 等可适当放宽）：当单表数据量预计超过 500 万行、或单库 QPS 预计超过 2000、或单表数据文件超过 10GB 时，应考虑分库分表。不满足上述任一条件时，跳过此步骤并在文档中说明"当前数据规模不需要分库分表"。
    *   如需要，设计分片键、分片策略和数据路由规则。
7.  **设计数据迁移方案**（如涉及）：
    *   **判断标准**：当项目涉及已有系统的数据迁移、或从旧表结构迁移到新表结构时，需要设计迁移方案。全新项目（无历史数据）跳过此步骤并在文档中说明"全新项目，无需数据迁移"。
    *   如涉及已有系统的数据迁移，设计迁移方案和回滚策略。
8.  **慢查询预防策略**：
    *   基于业务查询场景，给出索引优化建议。
    *   定义查询优化规则（如：避免 SELECT *、大表分页优化）。
9.  **生成文档**：确认无误后，生成最终的 Markdown 文档。
    *   读取 `templates/database-design-template.md` 作为生成基准。
    *   按模板填充所有章节。
    *   保存后向用户展示文档摘要并确认交付。

### 交互准则

- **性能意识**：每个索引的设计都必须基于具体的查询场景，避免冗余索引。
- **可扩展性**：表结构设计应预留扩展空间，但不过度设计。
- **最终交付**：详见 GUARDRAILS.md 全局交互准则第 7 条（保存路径：`specs/数据库设计.md`）。

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则方案不合格。

1. **ER 图必须完整**：所有核心实体及其关系必须在 ER 图中体现，不允许出现孤立实体。
2. **SQL 可直接执行**：所有建表 SQL 必须语法正确、可直接执行，不允许出现占位符或伪代码。
3. **每个表必须有注释**：表注释和字段注释必须完整，便于团队理解。
4. **索引必须有依据**：每个索引都必须标注其服务的查询场景，不允许出现无依据的索引。
5. **命名一致性**：表名、字段名必须遵循 `specs/数据模型字段约定.md` 中的约定，不得自行创造命名规则。
6. **跨文档一致性校验**：完成数据库设计后，必须执行 `python specs/validate_consistency.py --specs-dir specs/ --check field-names` 校验字段名一致性。如果校验结果包含 FAIL，必须修复不一致问题后才能继续。
   > 如果脚本不存在（如 `specs/validate_consistency.py` 未部署），则手动逐项比对数据模型字段约定与当前数据库设计的字段名一致性，确保一致。
7. **ORM 一对多关系的外键方向必须正确**：外键字段必须放在"多"的一方。不同 ORM 的语法不同（Prisma: `@relation(fields, references)` 放在外键方；SQLAlchemy: `ForeignKey()` 放在"多"方 + `relationship()` 在"一"方；Django ORM: `ForeignKey` 放在"多"方 + `related_name` 在"一"方；GORM: 外键 struct 嵌入"多"方；JPA: `@ManyToOne` 在"多"方 + `@OneToMany` 在"一"方；SeaORM: `Entity::belongs_to()` 放在"多"方 + `Entity::has_many()` 在"一"方（仅 Rust 项目适用）；EF Core: `[ForeignKey]` 放在"多"方 + `.HasOne()/.WithMany()` 在"一"方（仅 C# 项目适用）），但核心原则相同。
   - **自检方法**：生成 ORM 模型定义后，检查所有外键声明，确认其所在模型是"多"方（即字段类型不是列表/集合类型）。如果发现"一"方包含外键参数，必须将它们移到"多"方。
