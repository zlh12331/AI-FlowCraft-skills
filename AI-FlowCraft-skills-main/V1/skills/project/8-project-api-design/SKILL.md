## 指令

### 输入

- 用户的需求描述、想法或指令

- `templates/` 目录下的模板文件作为输出参考

### 你的任务

基于数据模型字段约定、系统架构设计和产品需求文档，设计完整的 API 接口文档（`specs/API接口文档.md`），包含所有接口的请求/响应格式、统一错误码、版本管理策略、限流规则和 AC 覆盖总表。API 设计完成的标准：文档已通过质量检查，所有核心接口均有完整的请求/响应定义，所有验收标准均有对应的 API 端点覆盖。

### 工作流程

1.  **读取上下文**：
    *   读取 `specs/数据模型字段约定.md`：了解字段命名、类型规范。
    *   读取 `specs/系统架构设计.md`：了解系统分层、服务划分和通信协议。
    *   读取 `specs/产品需求文档.md`：提取功能需求，反推所需的 API 端点。
2.  **设计接口列表总览**：
    *   梳理所有 API 端点，包含路径、HTTP 方法、简要说明和鉴权要求。
    *   标注每个接口对应的验收标准（AC）编号。
3.  **设计每个接口详情**：
    *   定义请求方法、路径、参数（Path/Query/Body）。
    *   定义响应格式（成功响应和异常响应）。
    *   定义鉴权要求和权限控制。
4.  **定义统一错误响应格式**：
    *   设计全局统一的错误响应结构。
    *   确保所有接口的错误响应格式一致。
5.  **定义统一错误码表**：
    *   设计错误码体系（业务错误码 + 系统错误码）。
    *   为每个错误码定义 HTTP 状态码、说明和处理建议。
6.  **定义接口版本管理策略**：
    *   设计版本号规则（如 URL 路径版本 / Header 版本）。
    *   定义版本升级和废弃策略。
7.  **定义限流规则**（如涉及）：
    *   **触发条件**：当以下任一条件满足时，需要设计限流规则——(1) 接口涉及公开访问（无需登录即可调用），(2) 接口涉及写操作且可能被批量调用（如注册、提交订单），(3) 接口调用成本较高（如涉及 AI 推理、第三方付费 API），(4) PRD 中明确提到高并发场景。不满足上述任一条件时，跳过此步骤并在文档中说明"当前接口无特殊限流需求"。
    *   设计接口级别的限流配置。
    *   定义限流算法和超限响应。
8.  **标注对应 AC 编号**：
    *   确保每个接口都关联到产品需求文档中的验收标准。
    *   对于通过查询参数间接覆盖的 AC（如筛选类 AC），也必须在接口列表总览中标注。
9.  **生成验收标准覆盖总表**：
    *   从 `specs/产品需求文档.md` 中提取完整的 AC 列表。
    *   如果 PRD 为复杂项目模式（功能独立文档存放在 `specs/features/` 目录下），还需读取该目录下所有功能独立文档，提取其中的 AC 条目，确保覆盖总表包含所有 AC。
    *   逐条检查每个 AC 是否都有对应的 API 端点覆盖。
    *   按模块分组生成覆盖总表，每个模块一个小节，末尾附覆盖统计表。
    *   如果某条 AC 无对应 API，标记为"未覆盖"并在文档中说明原因或补充缺失的接口。
10. **生成文档**：确认无误后，生成最终的 Markdown 文档。
    *   读取 `templates/api-design-template.md` 作为生成基准。
    *   按模板填充所有章节。
    *   保存后向用户展示文档摘要并确认交付。

### 文档质量保证

在生成文档后、保存前，**必须**按照 [../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md) 第 4 节「文档质量保证协议」依次执行两个阶段：

**阶段 A：深度完善循环** — 逐章审视内容深度、完整性、可执行性、示例与对比、上下文衔接，反复完善直到输出"文档已达到完全收敛状态"。

**阶段 B：Bug 检查循环** — 检查内部一致性、跨文档一致性、AC 覆盖、占位符残留、格式、逻辑、技术可行性，修复直到零 Bug。

> 这是进入下一阶段的硬性前提。未经两阶段检查的文档不得保存。

- **避免重复**：API 端点总表仅在本文档中维护。后端技术方案（Skill 10）应引用本文档而非重复列出端点表。如需在后端技术方案中提及 API，使用引用格式："完整 API 端点定义参见 `api-design.md`"。

### 输出模板

1. 检查 `specs/` 目录是否存在，若不存在请自动创建。
2. 读取 `templates/api-design-template.md` 作为生成基准。
3. 填好后保存为 `specs/API接口文档.md`。

### 交互准则

- **RESTful 规范**（仅 RESTful API 项目适用；GraphQL 项目参见 Skill 3 通信协议选项）：遵循 RESTful API 设计最佳实践，资源命名使用名词复数，HTTP 方法语义正确。
- **一致性**：所有接口的命名风格、响应格式、错误处理必须保持一致。
- **可追溯性**：每个接口都必须标注对应的 AC 编号，确保需求可追溯。
- **向后兼容**：接口设计应考虑向后兼容性，避免破坏性变更。
- **最终交付**：详见 GUARDRAILS.md 全局交互准则第 7 条（保存路径：`specs/API接口文档.md`）。

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则方案不合格。

1. **每个接口必须有完整的请求/响应示例**：不允许出现只有参数表而无示例的接口定义。
2. **每个接口必须标注 AC 编号**：不允许出现无 AC 关联的接口，确保需求可追溯。
3. **错误码必须完整**：每个接口的错误场景都必须有对应的错误码和错误提示。
4. **统一响应格式**：所有接口必须使用统一的响应格式，不允许出现格式不一致的情况。
5. **命名一致性**：接口路径、参数名必须遵循 `specs/数据模型字段约定.md` 中的约定，不得自行创造命名规则。
6. **跨文档一致性校验**：完成 API 设计后，必须执行 `python specs/validate_consistency.py --specs-dir specs/ --check field-names` 校验字段名一致性，以及 `python specs/validate_consistency.py --specs-dir specs/ --check api-paths` 校验 API 路径一致性（如果后端技术方案已存在）。如果校验结果包含 FAIL，必须修复不一致问题后才能继续。如果脚本不存在（如 `specs/validate_consistency.py` 未部署），则手动逐项比对上游文档与当前文档的字段名/路径一致性，将比对结果记录在文档末尾的"一致性校验记录"章节中。
