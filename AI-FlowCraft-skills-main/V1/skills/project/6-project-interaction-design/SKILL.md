## 指令

### 输入

- 用户的需求描述、想法或指令
- 项目相关文档（如已有 PRD、技术方案等）
- `templates/` 目录下的模板文件作为输出参考

### 你的任务

基于信息架构图和产品需求文档，设计完整的交互设计文档（`specs/交互设计文档.md`），涵盖页面流转、状态说明、异常处理和表单验证规则。交互设计完成的标准：文档已通过质量检查，所有核心页面均有完整的页面流转和异常处理定义。

### 工作流程

1.  **读取上下文**：
    *   读取 `specs/信息架构图.md`：了解页面层级和导航结构。
    *   **一致性检查**：读取信息架构图后，确认本 Skill 的页面定义与信息架构图中的页面清单一致。如信息架构图定义了内联交互/弹窗，本 Skill 应遵循相同的设计方式，不得擅自拆分为独立页面。
    *   读取 `specs/产品需求文档.md`：提取功能需求、用户故事和验收标准。
2.  **定义交互设计概述**：
    *   定义交互设计原则（一致性、即时反馈、容错性等），确保全局交互风格统一。
    *   概述全局交互规范：导航方式（如顶部导航/侧边栏）、反馈形式（如 Toast/弹窗/内联提示）、操作确认机制（如删除前二次确认）。
    *   逐页设计交互详述：为每个核心页面定义页面布局（区域划分、组件排列、视觉优先级）、页面概述、主要操作流程、状态切换（空态/加载态/错误态等）和反馈方式。
3.  **设计页面流转图**：
    *   梳理用户操作的完整路径，使用 Mermaid 流程图表达。
    *   覆盖主流程和关键分支流程。
4.  **设计各页面布局**：
    *   为每个核心页面定义页面布局，包括：区域划分（头部/主体/侧边栏/底部等）、组件排列顺序、视觉优先级（最重要的信息放在最醒目的位置）。
    *   使用 ASCII 图或文字描述表达布局结构，确保前端开发者可直接据此编码。
    *   响应式页面必须分别描述桌面端和移动端的布局差异。
5.  **定义状态说明**：
    *   定义全局状态类型：空态、加载态、错误态、成功态、禁用态。
    *   为每种状态指定视觉表现、触发条件和处理方式。
6.  **设计异常场景处理**：
    *   覆盖网络异常、数据为空、权限不足、服务端错误等场景。
    *   明确用户提示文案和恢复操作。
7.  **设计动效说明**（如涉及）：
    *   定义复杂动画的时序、缓动曲线、持续时间。
    *   仅在涉及复杂动画时填写，无动效需求则跳过。
8.  **定义表单验证规则**：
    *   区分实时校验（输入时）和提交校验（点击提交时）。
    *   为每个表单字段定义校验规则和错误提示文案。
9.  **生成文档**：确认无误后，生成最终的 Markdown 文档。
    *   读取 `templates/interaction-design-template.md` 作为生成基准。
    *   按模板填充所有章节。
    *   保存后向用户展示文档摘要并确认交付。

### 文档质量保证

在生成文档后、保存前，**必须**按照 [../../../specs/GUARDRAILS.md](../../../specs/GUARDRAILS.md) 第 4 节「文档质量保证协议」依次执行两个阶段：

**阶段 A：深度完善循环** — 逐章审视内容深度、完整性、可执行性、示例与对比、上下文衔接，反复完善直到输出"文档已达到完全收敛状态"。

**阶段 B：Bug 检查循环** — 检查内部一致性、跨文档一致性、AC 覆盖、占位符残留、格式、逻辑、技术可行性，修复直到零 Bug。

> 这是进入下一阶段的硬性前提。未经两阶段检查的文档不得保存。

### 交互准则

- **一致性**：所有页面的交互模式保持统一，同类操作使用相同的反馈方式。
- **完整性**：每个用户操作都必须有明确的反馈。
- **容错性**：异常场景的处理必须友好，提供明确的恢复路径，而非仅展示技术错误信息。
- **可验证性**：交互设计中的每个决策必须可追溯到产品需求文档中的功能需求或验收标准。
- **最终交付**：详见 GUARDRAILS.md 全局交互准则第 7 条（保存路径：`specs/交互设计文档.md`）。

### 底线规则

> 以下规则补充全局底线规则（见 GUARDRAILS.md），仅定义本 Skill 特有的约束。任何一条不满足则方案不合格。

1. **每个页面必须有状态说明**：不允许出现未定义状态的页面，至少覆盖空态、加载态和错误态。
2. **每个页面必须有布局描述**：不允许出现未定义布局的页面，必须描述区域划分、组件排列和视觉优先级，前端开发者可直接据此编码。
3. **每个异常场景必须有恢复方式**：不允许出现只展示错误信息但不提供恢复操作的场景。
4. **表单验证必须完整**：每个表单字段都必须定义校验规则和错误提示，不允许出现"无校验"的输入字段。
5. **页面流转必须覆盖主流程和关键分支**：不允许只描述正常流程而忽略异常分支。
6. **响应式布局必须包含移动端交互差异说明**：如果 PRD 或系统架构设计要求响应式布局，交互设计文档必须包含移动端与桌面端的交互差异说明（如：导航方式、操作手势、布局适配、表单输入方式等）。
7. **关键交互必须定义无障碍行为**：涉及键盘导航、焦点管理、屏幕阅读器的交互场景（如模态框、下拉菜单、表单错误提示），必须定义 ARIA 标注和键盘操作方式。
